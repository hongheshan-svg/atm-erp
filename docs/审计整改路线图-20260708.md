## 三、遗留平台级事项路线图（需立项，非一轮可完成）

以下事项审计定级 P2/P3，本质是**多周至多月的平台工程**，不是缺陷修复。逐项给出：差距、工作量级、分阶段方案、起步脚手架位置。

### R1. 多组织 / 多公司 / 多工厂（审计 ①⑦⑧⑨，评分 0/5）
- **差距**：`BaseModel` 无 org/company/factory 维度；SystemConfig 单例账套；库存/生产无 plant 隔离。
- **量级**：★★★★★（平台重构，全表加列 + 全查询过滤 + 全接口 + 数据迁移）。
- **分阶段**：
  1. 引入 `Organization/Company/Factory` 主数据 + `LegalEntity`。
  2. `BaseModel` 增 `company_id`（+ 可空 `factory_id`），默认管理器注入 `company` 过滤（类比现有 `SoftDeleteManager` 软删过滤，是现成的注入点）。
  3. 中间件/线程本地存当前 org 上下文；DataScope 扩展「本组织」范围。
  4. 分域灰度：先 masterdata/inventory/finance，再 sales/purchase/production。
  5. 公司间往来 + 合并报表维度。
- **起步脚手架**：复用 `apps/core/models.py:SoftDeleteManager` 的 QuerySet 注入模式，做 `CompanyScopedManager`；先在一个低耦合 app（如 oa 资产）验证端到端。

### R2. 国际化 / 海外部署（审计 ⑩，评分 0.5/5）
- **差距**：后端无 LocaleMiddleware/gettext/locale；前端零 vue-i18n；时区硬编码 Asia/Shanghai；多币种仅财务内半套。
- **量级**：★★★★（后端脚手架小，全量翻译 + 前端 i18n + 多时区 + 端到端多币种大）。
- **分阶段**：
  1. 后端：settings 加 `LocaleMiddleware`、`LANGUAGES`、`locale/` 目录；verbose_name 逐步包 `gettext_lazy`（可脚本半自动）。
  2. 前端：装 `vue-i18n`，抽取文案到 `locales/zh-CN.json`（可用扫描脚本），组件改 `$t()`。
  3. 多时区：用户/组织时区字段，`USE_TZ=True` 存 UTC、展示层转换。
  4. 多币种贯穿：销售/采购/库存金额加 currency + 记账时锁定汇率（财务已有 Currency/ExchangeRateHistory 可复用）。
- **起步脚手架**：财务 `Currency`/`ExchangeRateHistory` 已具备，作为多币种主数据底座。

### R3. 真 AI 能力（审计第十部分，AI 成熟度 ~4%）
- **差距**：零 LLM/向量/RAG/Agent/MCP；所有「智能」是规则/统计。
- **量级**：★★★★（从 0 到 1 的新子系统）。
- **分阶段（从最贴合现有数据的场景起步）**：
  1. **LLM 网关**：`apps/ai/` 新 app，统一多模型路由 + 调用审计 + 成本/限流（依赖：`anthropic`/`openai` 等，需入 requirements）。
  2. **知识库 RAG**：先把 `projects/knowledge_models` 的 SQL LIKE 检索升级为 Elasticsearch（ES 已部署）→ 再加向量列（pgvector 或独立向量库）+ embedding + 检索增强。
  3. **报价 Copilot**：以现有 `quote_estimation` 成本模型为工具，LLM 辅助生成报价说明/风险提示（人审）。
  4. **Agent + MCP 工具层**：把 440+ 业务模型的读接口封装为工具，做审批摘要/经营问答（NL2SQL 走 `dashboard_config` 已有的只读 SQL 安全层）。
  5. 排产/预测换真模型（OR-Tools / 时序），替换现贪心/规则。
- **起步脚手架**：`core/dashboard_config.py` 的只读 SQL 安全校验层可作为 NL2SQL 的执行沙箱；ES 已接入可作 RAG 检索层。

### R4. IPD 阶段门（审计第四部分，成熟度 ~30%）
- **差距**：无 Charter/DCP、无 ES/EVT/DVT/PVT/MP 阶段门、无 MRD/PRD 区分、无配置管理。
- **量级**：★★★（新流程模型，可复用现有评审引擎）。
- **分阶段**：
  1. `StageGate` 模型（阶段 + go/kill 决策 + 决策评审 DCP/TR），复用 `proposal.py`/`requirement_review.py` 的评审+打分机制。
  2. 需求分级 MRD/PRD（Requirement 加 doc_type）。
  3. 项目组合/路线图管理。
- **起步脚手架**：`requirement_review`/`proposal` 的加权评审模板直接复用为阶段门决策评审。

### R5. EBOM/MBOM 分离 + 结构化 BOM 对比（审计第四部分）
- **差距**：单一 `ProjectBOM`；`BOMVersion` 仅整表快照；对比按 item_id 忽略层级。
- **量级**：★★★。
- **分阶段**：BOM 加 `bom_type`（DESIGN/ENG/MFG）+ 父子结构（多级）；对比算法改为按结构树 diff；生效日期/替代料已有基础（`bom_advanced.BOMSubstitute`）。

### R6. 生产六套工单收敛 + 工艺→报工→成本主链（审计第六部分，生产 50%/MES 40%）
- **差距**：6 套并行工单模型（ProductionPlan/ScheduleOrder/ProductionSchedule/FiniteCapacityPlan/WorkOrder/ProjectRouting），两套共用 'SCH' 前缀；APS 是玩具（不读工艺、工时=数量×1、策略字段是摆设）。
- **量级**：★★★★★（需推倒重做而非修补）。
- **分阶段**：
  1. 选定单一「生产工单」聚合根，其余降为视图/子表；统一编号。
  2. 工艺路线 standard_hours 驱动排产；报工产生人工成本凭证（工时×费率）。
  3. APS 引入 OR-Tools 有限产能求解器（班次日历 + 设备能力约束 + 正/倒排）。
- **起步脚手架**：`routing.py` 工时标准 + `advanced_cost_tracking.LaborRateStandard` 费率库，是「报工进成本」的现成两端，只差接线。

### R7. 服务层统一 + core/projects 巨石拆分（审计第十四部分）
- **差距**：无一致 service 层（23 文件命名混乱），业务逻辑淤积 2000–3700 行胖 views + 11 个含事务的 serializer；core（39 模型）/projects（157 模型）巨石化。
- **量级**：★★★★（持续重构）。
- **分阶段**：定义 service 层规约（业务逻辑出 views/serializers）；`payable_adapters` 的适配器模式是全库范本，推广之；按限界上下文拆 projects（field_service/equipment/plm 独立）。

### R8. 事件总线 + 插件机制（审计第十七部分，为微服务/生态留口）
- **差距**：无领域事件（仅 signals + 适配器模拟）、无插件注册、模块间直接 import 成环。
- **量级**：★★★★。
- **分阶段**：先做依赖治理（跨 app 只经 service 接口，禁止反向 import）→ 引入领域事件（可先在进程内 event bus）→ 插件注册（entry_point/hook）→ 具备拆分条件后再谈微服务。

1. **已消除**：全部 P0 账实/资金/崩溃缺陷 + 关键 P1 闭环与安全项 —— 可支撑单公司试点。
