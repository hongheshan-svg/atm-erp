# 《非标自动化 AI ERP 项目专业审计报告》

**审计日期**：2026-07-08
**审计方式**：全代码库只读审计。11 个专项审计组并行深查（架构 / 销售CRM / 采购SRM / 库存MRP / 生产MES / 项目PLM·IPD / 财务成本 / AI能力 / 权限安全 / 前端OA / 数据库·代码质量），全部结论附 file:line 证据，可逐条复核。
**系统基线**：后端 Django 4.2，生产代码约 120,620 行 Python（不含迁移/测试），479 个模型、423 个 ViewSet、1113 个自定义 @action；前端 Vue3+TS，206 个 .vue 页面、198 条路由；微信小程序 10 页；485 次提交。
**对标基准**：SAP S/4HANA (SD/MM/PP/FI/CO/WM)、Oracle ERP、Microsoft Dynamics 365、Infor、IFS；华为 IPD；西门子 Opcenter (MES) / Teamcenter (PLM)；Ariba (SRM)。
**利益声明**：按委托要求，不因开发者身份调高评分，发现问题直接指出。

---

## 执行摘要

1. **这是一套模块覆盖面惊人、行业理解真实的传统单体 ERP**——面向单法人、单工厂、纯国内中文场景。非标自动化行业的交付链（技术协议、FAT/SAT、12 类调试记录、3D 装配指导、分期收款含质保金、按项目 BOM 驱动 MRP）是全系统最扎实的部分，说明作者确实懂这个行业。
2. **"AI Native ERP" 定位不成立**。全仓库零 LLM 调用、零向量库、零 RAG、零 Agent、零 MCP，requirements.txt 中没有任何 AI 依赖。所有"智能"命名的功能实为 SQL 聚合、if/else 阈值规则、成本加成公式和贪心排序。AI 成熟度约 4%，连 "AI Enabled" 都算不上。
3. **广度换深度的代价严重**。审计确认 6 处 P0 级账实/资金正确性缺陷（退料双倍入库、收付款冲销不回退、可过账不平衡凭证、信用管控被审批流绕过等），以及至少 8 处"引用不存在的字段/模型"导致的运行期崩溃或静默失效——比价、催货任务、项目健康度、BOM 成本卷积、WIP 看板均属此类。多个核心引擎（APS、总账自动过账、成本自动归集、会签）是"有模型、无引擎"的空壳。
4. **上线判断**：修复 P0 后可支撑单公司、行政+项目+采购+库存+往来账层面的试点使用；财务核算（总账/报表）、生产排产、质量管理、成本核算四个域当前不具备上线条件。多组织、国际化、AI 属于架构级缺失，非增量可补。

# 第十一部分：数据库设计审核

**评分 6.2/10。** 地基规范（统一 BaseModel/软删/审计、主数据 PROTECT、全局强制分页），最关键的库存扣减把事务+悲观锁+DB 约束三件事做对了；但正确实践未系统化推广：

- **索引欠账（P1）**：479 个模型仅 ~6.5% 有显式复合索引；is_deleted 全局无索引却出现在每条查询；最高频增长的 StockMove 无任何业务索引（move_date/status/reference 均裸奔）；AR/AP 排序与逾期扫描列无索引；SystemNotification 无 user+is_read 复合索引。
- **唯一约束 × 软删除冲突（P2）**：100 个业务编号字段用全局 unique=True（含软删行），条件唯一约束全系统仅 1 处（Stock）；AR 编号自增查询排除软删行但约束包含软删行，复用即 IntegrityError。
- **并发竞态（P1）**：AR/AP 单号用"查最大号+1"内联生成，无锁无 atomic（finance/models.py:236-255），并发撞号——而系统已有带锁的 code_rule 生成器却没用。
- **金额精度双标（P2）**：max_digits 18/15/12/10 并存，finance 内部同时用 18(102 处)与 15(197 处)，缺统一 MoneyField。
- **on_delete 语义错配（P3）**：软删除模型上挂 360 个物理 CASCADE。
- **无字段级历史（P2）**：零 simple_history/reversion；审计日志 90 天物理硬删——对财务合规（通常要求数年）不足，且单次海量 delete 会长事务锁表。
- **缓存（良好）**：权限缓存有精准失效策略（delete_pattern）。
- **分库分表**：无，当前量级不必须，但大表无归档策略。

# 第十六部分：安全审核

| 项 | 结论 |
|----|------|
| SQL 注入 | 基本安全（ORM 为主，custom_sql 用参数绑定）；但仪表盘 custom_sql 黑名单式过滤本身允许**任意表 SELECT**（含 user 表口令哈希），且 DashboardWidgetViewSet 无 IsSystemAdmin 硬门槛（dashboard_config.py:409-469）——内部人员可导出全员 PBKDF2 哈希 |

# 问题清单

## 【P0】必须立即修复（账实/资金正确性与真实数据下的崩溃）

1. **退料入库双倍计数**——先 StockMove 入库再手工 +=，每笔良品退料库存翻倍，连锁污染加权平均成本（inventory/material_views.py:365-389）。
2. **收付款删除/作废不回退 AR/AP 已收付额**——Payment 只在新建时 F() 累加，无反向逻辑；红冲后应收应付虚减（finance/models.py:443-459）。
3. **凭证可不平衡过账、可向已关闭期间过账**——approve/post 不校验 is_balanced 与期间状态，且 PENDING 态还能追加分录（finance/accounting.py:567,603,617）。
4. **信用管控被审批流绕过**——走 SALES_ORDER 审批的订单在回写处只建应收不做信用校验，冻结/黑名单/超额客户经审批即成交（core/workflow/services.py:434-446）。
5. **采购比价在真实数据下必崩**——auto_score_reliability 引用 PO 不存在的状态与字段，供应商有历史完成单即整事务回滚（purchase/comparison_service.py:313,327-328）。
6. **"字段幻觉"类运行期崩溃/静默失效清单**（同一根因：无回归测试防护）：交期催货任务每日崩（purchase/tasks.py:51-89，已在 beat 排程）；项目健康度 due_date/DONE/progress 三处字段不存在（health_dashboard.py:137-195）；BOM 成本卷积 import 不存在的 BOM 模型（bom_cost_rollup.py:94）；BOM 版本快照/对比引用不存在的 unit_cost（bom_advanced.py:160,519）；WIP 看板 import 不存在的 ProductionOrder（kanban_wip.py:64）；里程碑进度回写不存在的 project.progress（milestone.py:481）；报价估算转正式报价字段全错（quote_estimation.py:759-785）；看板质检枚举 PASSED≠PASS 合格率恒 0（kanban.py:132）。

# 总结论

本系统的真实身份是：**一套由深刻理解非标自动化行业的团队开发的、覆盖 26+ 业务域的传统 Django 单体 ERP**。它的行业 Know-how（技术协议、FAT/SAT、调试管理、分期收款、项目驱动 MRP）是许多标准 ERP 产品做不到的；它在关键并发场景（库存扣减、应付核销、审批流转）表现出真实的工程能力；TypeScript 前端与考勤真机对接说明团队执行力不弱。

它的核心病灶只有一个：**广度扩张的速度远超质量防护网的建设速度**。5.3% 的测试比率无法支撑 12 万行代码 479 个模型的演进，于是出现了大量"引用不存在字段/模型"的幻觉代码、"有模型无引擎"的空壳功能和 6 套并行工单这类失控的重复建设。2026-06 的 222 项整改之后本次审计仍发现新的 P0，证明问题是系统性的。

**三条战略建议**：
1. **先止血，再扩张**：冻结新功能，集中修复 P0 与 P1 业务闭环，同时把 CI 测试 gate 建起来——否则修复速度永远追不上劣化速度。
2. **诚实定位**：以"非标自动化行业 ERP"为卖点（这是真差异化），摘掉"AI Native"标签直到第一个真 LLM 场景落地；AI 路线建议从"知识库 RAG + 报价 Copilot"两个最贴合现有数据的场景起步。
3. **做减法**：合并双轨/多轨模型（工单、RFQ、ECN、通知、收款、售后），系统的下限不取决于它有多少功能，取决于最不可信的那条数据链。
