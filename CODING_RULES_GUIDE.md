# 编码规则使用指南

## 概述

ERP系统现已支持灵活的编码规则管理，系统管理员可以在"系统管理 > 编码规则"页面自定义各种业务单据的编号规则。

## 功能特性

### 1. 支持的编码类型

- **项目编号** (PROJECT)
- **物料编码** - **注意**：物料编码有特殊规则，不在此系统配置，请在"基础数据-物料管理"页面使用编码生成功能
- **采购合同** (PURCHASE_CONTRACT)
- **销售合同** (SALES_CONTRACT)
- **采购申请** (PURCHASE_REQUEST)
- **采购订单** (PURCHASE_ORDER)
- **销售订单** (SALES_ORDER)
- **销售报价** (SALES_QUOTE)
- **发货单** (DELIVERY_ORDER)
- **发票** (INVOICE)
- **收货单** (GOODS_RECEIPT)
- **库存移动** (STOCK_MOVE)
- **库存调整** (STOCK_ADJUSTMENT)

### 2. 编码格式配置

每个编码规则可以配置以下元素：

#### 固定前缀
- 编码的固定前缀部分
- 例如：PRJ、ITEM、PC、SC等
- 最大长度：20个字符

#### 日期格式
支持的日期标记：
- `YYYY` - 4位年份（如：2025）
- `YY` - 2位年份（如：25）
- `MM` - 2位月份（如：01-12）
- `DD` - 2位日期（如：01-31）

可自由组合，例如：
- `YYYYMMDD` → 20251225
- `YYYY-MM` → 2025-12
- `YYMMDD` → 251225

#### 序列号
- **长度**：1-10位，不足位数时前面补0
- **起始值**：序列号从哪个数字开始
- **当前值**：当前已使用到的序列号

#### 重置模式
- **不重置** (NONE)：序列号持续递增，永不重置
- **每日重置** (DAILY)：每天重置为起始值
- **每月重置** (MONTHLY)：每月重置为起始值
- **每年重置** (YEARLY)：每年重置为起始值

#### 分隔符
- 各部分之间的分隔符
- 常用：`-`、`_`、`.`等
- 留空表示无分隔符

### 3. 编码示例

#### 示例1：项目编号
```
前缀：PRJ
日期格式：YYYYMM
序列号长度：4
序列号起始：1
重置模式：每年重置
分隔符：-

生成示例：PRJ-202512-0001
```

## 使用方法

### 初始化默认规则

1. 登录系统，进入"系统管理 > 编码规则"
2. 点击"初始化默认规则"按钮
3. 系统会自动创建常用的编码规则（已存在的规则不会被覆盖）

### 添加新规则

1. 点击"添加规则"按钮
2. 选择规则类型（每种类型只能有一个规则）
3. 配置编码格式各项参数
4. 查看示例预览，确认格式正确
5. 点击"保存"

### 编辑规则

1. 找到要编辑的规则，点击"编辑"按钮
2. 修改配置参数
3. 注意：规则类型不可修改
4. 点击"保存"

### 重置序列号

如果需要手动重置某个规则的序列号：
1. 找到对应规则，点击"重置序号"按钮
2. 确认操作
3. 序列号将重置为起始值

### 查看历史记录

1. 点击规则的"历史"按钮
2. 可以查看该规则生成的所有编码历史
3. 包括：生成的编码、序列号、生成人、生成时间等信息

### 启用/停用规则

1. 编辑规则时，可以切换"是否启用"开关
2. 停用的规则不会被使用，系统会回退到默认编码规则

## 技术说明

### 后端集成

在业务模型中使用编码规则：

```python
from apps.core.utils import generate_code

class Project(BaseModel):
    code = models.CharField(max_length=50, unique=True, blank=True, verbose_name='项目编号')
    
    def save(self, *args, **kwargs):
        if not self.code:
            # 使用编码规则生成项目编号
            self.code = generate_code('PRJ', rule_type='PROJECT')
        super().save(*args, **kwargs)
```

### API接口

- **列表查询**：`GET /api/core/code-rules/`
- **创建规则**：`POST /api/core/code-rules/`
- **更新规则**：`PUT /api/core/code-rules/{id}/`
- **删除规则**：`DELETE /api/core/code-rules/{id}/`
- **重置序号**：`POST /api/core/code-rules/{id}/reset_sequence/`
- **查看历史**：`GET /api/core/code-rules/{id}/history/`
- **初始化默认规则**：`POST /api/core/code-rules/init_default_rules/`

### 权限控制

编码规则管理功能仅对**系统管理员**开放。

## 注意事项

1. **唯一性**：每种规则类型只能有一个活动的编码规则
2. **物料编码特殊性**：物料编码支持手动输入，如果手动输入了编码，则不会自动生成
3. **序列号重置**：重置序列号是危险操作，可能导致编码重复，请谨慎使用
4. **历史记录**：编码生成历史会自动记录，用于审计和追踪
5. **线程安全**：编码生成使用数据库行锁，确保并发环境下的唯一性
6. **回退机制**：如果编码规则不可用，系统会自动回退到默认的编码生成规则

## 常见问题

### Q: 修改编码规则后，已有的编码会改变吗？
A: 不会。编码规则只影响新生成的编码，已有的编码不会改变。

### Q: 可以删除正在使用的编码规则吗？
A: 可以删除，但删除后系统会回退到默认编码规则。建议先停用规则，确认无影响后再删除。

### Q: 序列号用完了怎么办？
A: 如果序列号达到最大值（由长度决定），建议：
   1. 增加序列号长度
   2. 或者修改日期格式，增加时间维度
   3. 或者调整重置模式

### Q: 不同项目可以使用不同的编码规则吗？
A: 当前版本不支持。每种类型全局只有一个编码规则。如有此需求，可以通过前缀或其他方式区分。

### Q: 如何确保编码不重复？
A: 系统使用数据库行锁和事务机制，确保并发环境下编码的唯一性。同时，编码字段在数据库层面设置了唯一约束。
